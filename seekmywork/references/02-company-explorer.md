# 公司遍历与筛选

## Overview

根据用户的行业、专业和目标城市，系统性地发现、筛选、调研目标公司。采用 OSINT（开源情报）方法论，从多维度构建公司画像，输出结构化的推荐清单。

**核心价值：** 在用户盲目投递之前，先帮他锁定一批"值得投"的公司，避免海投浪费时间。

**参考来源：**
- [LessieAI/lessie-skill](https://github.com/LessieAI/lessie-skill) — 公司搜索、筛选、丰富管线
- [mtjikuzu/hermes-osint-skill](https://github.com/mtjikuzu/hermes-osint-skill) — OSINT 尽职调查流程
- [ScaleBrick/founder-marketing-skills](https://github.com/ScaleBrick/founder-marketing-skills) — 竞品分析框架
- [MichaelRochonnn/buffett-investment-research](https://github.com/MichaelRochonnn/buffett-investment-research) — 结构化公司分析

## When to Use

- ✅ Module 1（简历生成）完成后，用户需要确定目标公司
- ✅ 用户说"帮我找找长沙有哪些好公司"
- ✅ 用户想了解某行业在某城市的公司分布
- ✅ 用户想对比多家公司的口碑
- ✅ 用户想了解某公司的校招情况

### 不适用场景
- ❌ 用户已明确指定目标公司（但仍需做 Phase 3 背调）
- ❌ 用户只需要校招网站链接（→ Module 3）
- ❌ 用户没有简历数据（提示先执行 Module 1）

## Core Process

### Stage 0: 筛选条件采集

从用户输入或上游模块继承以下信息：

| 信息 | 必填 | 来源 |
|------|------|------|
| 目标行业 | ✅ | 用户指定（如"网络安全""人工智能""互联网"） |
| 专业背景 | ✅ | Module 1 简历数据（如"信息安全""计算机科学"） |
| 目标城市 | ✅ | 用户指定（如"长沙""深圳"） |
| 目标公司 | 选填 | 用户指定（如有则优先，但仍做背调） |
| 公司规模偏好 | 选填 | 大厂/中厂/创业公司/不限 |
| 岗位方向 | 选填 | Module 1 简历数据（如"安全工程师""后端开发"） |

**信息采集规则：**
- 一次只问一个问题
- 如果用户在 Module 1 已提供部分信息，直接继承，不重复询问
- 如果用户说"不限"，按行业主流公司覆盖

### GATE 0 — 条件完整性

- [ ] 目标行业已明确
- [ ] 专业背景已确认
- [ ] 目标城市已确认

### Stage 1: 公司发现

#### 1.1 已知公司库匹配

首先检查用户工作目录下是否存在 `company-db.md`（本地公司知识库）。如果存在，从中查找匹配的公司。如果不存在，跳过本地库匹配，直接进入网络搜索。

#### 1.2 网络搜索发现

使用 `web_search` 按以下策略搜索，**每轮至少 3 个不同查询词**：

**搜索策略（按优先级）：**

| 轮次 | 查询模式 | 目的 |
|------|---------|------|
| 1 | `{城市} {行业} 公司 排名 2026` | 行业排名 |
| 2 | `{城市} {行业} 产业园 企业 入驻` | 产业园区 |
| 3 | `{城市} {行业} 招聘 公司 校招` | 招聘活跃公司 |
| 4 | `{行业} {城市} 企业名录 融资` | 融资公司 |
| 5 | `site:kanzhun.com {城市} {行业}` | 看准网公司列表 |
| 6 | `{城市} {行业} 独角兽 创业公司` | 新兴公司 |

**并行搜索**：使用 `delegate_task` 的 tasks 批量模式，每批最多 3 个查询。

#### 1.3 搜索结果提取

从每个搜索结果中提取：

```json
{
  "name": "公司名称",
  "industry": "所属行业",
  "scale": "公司规模（人数）",
  "type": "本土上市/分支/创业/外资",
  "hq_city": "总部城市",
  "is_local": true,
  "main_business": "主营业务一句话",
  "source_url": "信息来源链接"
}
```

**提取规则：**
- 优先提取本土总部公司（HQ 在目标城市）
- 其次提取在目标城市有研发中心的大厂
- 排除与用户专业完全不匹配的公司
- 每条信息必须附来源 URL

#### 1.4 去重与合并

合并来自不同搜索源的同一家公司信息，去重后生成候选列表。

### GATE 1 — 公司发现质量

- [ ] 覆盖至少 3 个搜索查询词
- [ ] 候选公司不少于 15 家
- [ ] 每家公司有来源 URL

### Stage 2: 公司画像构建（OSINT）

对候选公司（Top 15-20 家）构建结构化公司画像。

#### 2.1 基础信息采集

对每家公司，从公开信息中提取：

```json
{
  "name": "公司名称",
  "profile": {
    "industry": "行业",
    "sub_industry": "细分领域",
    "founded": "成立年份",
    "scale": "员工规模",
    "type": "上市/私营/国企/外资",
    "stock_code": "股票代码（如有）",
    "funding_stage": "融资阶段（如有）",
    "hq_city": "总部城市",
    "branches": ["分支机构城市列表"],
    "main_products": ["核心产品/服务"],
    "tech_stack": ["技术栈（从 JD/官网提取）"],
    "website": "官网 URL",
    "careers_url": "招聘页面 URL"
  }
}
```

**数据来源：**
- 公司官网「关于我们」页面
- 天眼查/企查查（公司基本信息）
- 招聘页面（技术栈、在招岗位）
- 新闻报道（融资、业务动态）

#### 2.2 竞品分析（同一行业内的公司对比）

对同一细分领域的公司进行横向对比：

```markdown
### 竞品对比：{细分领域}

| 维度 | 公司A | 公司B | 公司C |
|------|-------|-------|-------|
| 规模 | 5000+ | 1000+ | 200+ |
| 核心产品 | 产品X | 产品Y | 产品Z |
| 技术栈 | Java/Go | Python/C++ | Rust/Go |
| 融资 | 上市 | B轮 | A轮 |
| 薪资水平 | 中上 | 中等 | 偏高 |
| 加班强度 | 中 | 高 | 低 |
```

#### 2.3 风险评估

对每家公司进行风险扫描：

| 风险类型 | 检查内容 | 数据来源 |
|---------|---------|---------|
| 裁员风险 | 近 6 个月是否有裁员新闻 | web_search |
| 经营风险 | 是否有欠薪/倒闭/诉讼新闻 | web_search |
| 负面舆情 | 员工大规模吐槽/维权 | 脉脉、知乎 |
| 行业风险 | 所在行业是否下行 | 行业报告 |
| 竞争风险 | 是否面临激烈竞争/被替代 | 行业分析 |

**风险等级：**
- 🟢 低风险：无负面信号
- 🟡 中风险：有少量负面信号但不影响整体
- 🔴 高风险：有裁员/欠薪/大规模负面评价

### GATE 2 — 公司画像质量

- [ ] 每家公司有完整的 profile JSON
- [ ] 同行业公司有竞品对比
- [ ] 每家公司有风险评估

### Stage 3: 员工评价调研

对 Top 10-15 家公司进行员工评价调研。

#### 3.1 评价数据源

| 平台 | 搜索词 | 提取维度 |
|------|--------|---------|
| 看准网 | `site:kanzhun.com {公司名} 怎么样` | 综合评分、薪资、面试 |
| 脉脉 | `site:maimai.cn {公司名} 评价` | 工作氛围、加班、晋升 |
| 知乎 | `site:zhihu.com {公司名} 工作 体验` | 深度评价、优缺点 |
| 牛客 | `site:nowcoder.com {公司名} 面经` | 面试难度、面试流程 |
| 小红书 | `{公司名} 工作体验 评价` | 真实体验、日常感受 |

#### 3.2 评价维度提取

| 维度 | 说明 | 评分标准 |
|------|------|---------|
| 综合评分 | 看准网评分（如有） | 直接引用 |
| 工作氛围 | 团队/文化描述 | 正面/中性/负面 |
| 薪资福利 | 薪资范围、福利 | 高于/持平/低于行业 |
| 加班情况 | 工作时长 | 正常/偶尔/常态 |
| 晋升发展 | 晋升通道 | 明确/模糊/无 |
| 面试难度 | 面试流程 | 简单/中等/困难 |
| 校招友好度 | 是否有校招、实习转正 | 积极/一般/不友好 |

#### 3.3 评价可信度标注

| 等级 | 标准 |
|------|------|
| ⭐⭐⭐⭐⭐ | 多平台一致、评价数量 100+ |
| ⭐⭐⭐⭐ | 评价较多但有分歧 |
| ⭐⭐⭐ | 评价较少（10-50 条） |
| ⭐⭐ | 极少评价或信息矛盾 |
| ⭐ | 无法获取评价 |

### Stage 4: 综合评分

对每家公司从以下维度打分（1-5 分）：

| 维度 | 权重 | 评分依据 |
|------|------|---------|
| 行业匹配度 | 20% | 公司主营业务与目标行业的匹配程度 |
| 专业匹配度 | 20% | 公司技术栈与用户专业的匹配程度 |
| 员工口碑 | 15% | 员工评价综合得分 |
| 发展前景 | 15% | 融资阶段、行业地位、增长趋势 |
| 薪资竞争力 | 10% | 薪资水平（如有数据） |
| 校招友好度 | 10% | 是否有校招通道、实习转正率 |
| 风险评估 | 10% | 裁员/经营/舆情风险（反向评分） |

**综合得分** = 各维度加权得分

## Rules

### 必须遵守
- ✅ 公司遍历必须覆盖至少 3 个搜索查询词
- ✅ 员工评价必须注明来源平台，不可编造
- ✅ 评价中正面和负面都要如实记录，不可只报喜
- ✅ 综合评分必须有明确的维度和权重
- ✅ 推荐清单必须有分梯队的投递建议
- ✅ 每家公司必须有风险评估
- ✅ 竞品分析必须基于同一细分领域
- ✅ 公司画像必须有结构化 JSON 数据

### 禁止事项
- ❌ 编造员工评价或薪资数据
- ❌ 对公司做"好/坏"的价值评判，只呈现事实
- ❌ 推荐与用户专业完全不匹配的公司
- ❌ 忽略用户指定的目标公司（如有指定，必须纳入）
- ❌ 隐瞒已知的风险信号（裁员/欠薪等）
- ❌ 跳过竞品对比直接推荐

## Common Rationalizations

| 借口 | 现实 | 应对 |
|------|------|------|
| "这家公司在目标城市没有岗位" | 没有岗位也要告知用户 | 在清单中标注"暂无校招岗位" |
| "员工评价都是负面的" | 负面评价也是有价值的信息 | 如实记录，让用户自己判断 |
| "搜不到太多信息" | 信息少不代表可以编造 | 标注评价可信度为 ⭐，建议用户自行补充调研 |
| "用户已经指定了公司，跳过调研" | 指定公司也需要了解口碑 | 对指定公司也做评价调研 |
| "风险评估太敏感，不写了" | 用户需要知道风险才能做决定 | 如实标注，但不做价值判断 |
| "竞品分析太复杂，跳过" | 竞品对比帮助用户理解行业定位 | 至少做同一细分领域的 3 家对比 |

## Red Flags

- 🚩 搜索结果中公司大量裁员/倒闭新闻：在清单中标注 🔴 高风险
- 🚩 员工评价中频繁出现"拖欠工资""强制加班"等关键词：高风险标注
- 🚩 用户目标行业在目标城市几乎没有公司：建议扩大城市范围或调整行业
- 🚩 所有公司评分都很低：建议用户重新评估目标
- 🚩 公司官网无法访问或招聘信息过期：标注信息可信度低
