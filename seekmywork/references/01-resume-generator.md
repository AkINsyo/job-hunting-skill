# 简历生成

## Overview

通过对话采集用户信息，结合目标岗位 JD 分析，生成 ATS 可读的结构化简历。支持 Markdown + JSON 双格式输出。

## When to Use

- ✅ 用户没有简历，需要从零生成
- ✅ 用户有简历但格式不规范，需要结构化
- ✅ 用户想针对特定岗位优化简历
- ✅ 完整求职流程的第一步

## Core Process

### Stage 1 — 信息采集（Required）

**规则：一次只问一个问题。用户回答后再问下一个。**

按以下顺序采集：

**Q1 — 基本信息**
> 请先告诉我你的基本情况：学校、专业、学历（本科/硕士/博士）、毕业时间。

记录：学校名称、专业、学历层次、毕业年月。如实标注 985/211/双一流。

**Q2 — 目标岗位（可选但强烈建议）**
> 你目标的岗位方向是什么？有没有具体的 JD（职位描述）可以给我？有的话我可以帮你针对性优化。

如果有 JD → 进入 Stage 1a（JD 分析）。没有 → 跳过，按通用方向生成。

**Q3 — 技术/技能栈**
> 你会哪些技术/技能？按编程语言、框架工具、其他技能分类列一下。

按岗位类型动态调整技能维度：

| 岗位类型 | 技能维度 |
|---------|---------|
| 后端开发 | 编程语言、框架、数据库、中间件、云/DevOps |
| 前端开发 | 语言、框架、构建工具、CSS/UI、性能优化 |
| 安全/渗透 | 语言、安全工具、漏洞类型、合规标准、渗透方法 |
| 数据/AI | 语言、ML 框架、数据工具、可视化、统计方法 |
| 产品/运营 | 分析工具、原型工具、项目管理、行业知识 |
| 通用 | 编程语言、框架/工具、其他技能 |

**Q4 — 实习经历**
> 有没有实习经历？有的话告诉我：公司名、岗位、做了什么、持续多久。没有也没关系。

如果有实习 → 追问核心产出（量化数据）。

**Q5 — 项目经历**
> 做过哪些项目？项目名、用了什么技术、你负责什么、有什么成果/亮点。

每个项目追问：
- 具体负责什么模块？
- 有没有量化成果？（性能提升 X%、处理 X 条数据、服务 X 用户）
- 遇到了什么问题，怎么解决的？

**Q6 — 其他亮点（可选）**
> 有没有竞赛获奖、论文发表、开源贡献、技术博客？没有可以跳过。

**完成条件：** Q1 + Q3-Q5 全部回答完毕。向用户复述采集到的信息摘要，请用户确认。

### Stage 1a — JD 分析（Optional, 强烈建议）

当用户提供了目标岗位 JD 时执行。

从 JD 中提取：
1. **必备条件** vs **加分条件**（区分清楚）
2. **高频关键词**（重复出现的技能/工具/动词）
3. **岗位级别**（实习/初级/中级/高级）
4. **业务领域**（toB/toC/安全/数据/基础架构等）

输出摘要：
```
=== JD 分析 ===
目标岗位：XXX
岗位级别：初级/实习
必备条件：A, B, C
加分条件：D, E
ATS 关键词：X, Y, Z（需在简历中自然出现）
```

### GATE 1 — 信息完整性检查

- [ ] 学校、专业、学历、毕业时间已确认
- [ ] 至少有 1 项技能
- [ ] 至少有 1 个项目或实习
- [ ] 用户已确认信息摘要

未通过 → 继续采集，不进入下一阶段。

### Stage 2 — 简历策略

在写内容之前，先确定策略：

1. **简历版块顺序**（根据用户级别）：

| 级别 | 版块顺序 |
|------|---------|
| 实习/应届（无实习） | 教育 → 技能 → 项目 → 竞赛/荣誉 |
| 应届（有实习） | 教育 → 实习 → 技能 → 项目 → 竂赛/荣誉 |
| 有 1-2 年经验 | 实习/工作 → 技能 → 项目 → 教育 |

2. **重点突出项**：根据 JD 分析，确定 3 个核心优势
3. **弱化/省略项**：与目标无关的经历

### GATE 2 — 策略确认

- [ ] 版块顺序已确定
- [ ] 核心优势已识别

### Stage 3 — 内容撰写

#### 3a — 经历描述（STAR 法则 + 量化）

每条经历用 **STAR 法则** 撰写：

> **S**ituation（背景）→ **T**ask（任务）→ **A**ction（行动）→ **R**esult（成果）

**写作公式：**
> 动词 + 做了什么 + 用什么方法/工具 + 可量化结果 + 规模/上下文

**强动词库**（禁止用"负责""参与""协助"）：

| 类别 | 动词 |
|------|------|
| 开发 | 构建、开发、实现、设计、重构、优化、封装 |
| 分析 | 分析、诊断、定位、评估、调研、对比 |
| 管理 | 主导、推动、协调、规划、分配、跟进 |
| 成果 | 提升 X%、降低 X%、处理 X 条、服务 X 用户、缩短 X 天 |

**示例：**
- ❌ `负责后端开发`
- ✅ `独立开发用户认证模块，使用 Spring Boot + JWT，支持 5000+ 用户并发登录`

#### 3b — 技能呈现

按 JD 关键词调整技能顺序，匹配度高的排前面。

如有 JD → 在技能表中标注与 JD 的匹配度：
- 🟢 熟练匹配
- 🟡 了解/部分匹配
- （不标注不匹配的）

#### 3c — 人性化检查

对所有生成内容执行检查：

```
人性化检查：
[PASS] 无空洞套话（"精通""熟练掌握""良好的沟通能力"）
[PASS] 无连续 3 条以上相同句式开头
[PASS] 无 AI 典型表达（"具备丰富的""拥有扎实的"）
[PASS] 每条描述都能在 30 秒内自然口头表达
[PASS] 所有量化数据合理可信
```

未通过 → 重写不合格的内容。

### GATE 3 — 内容质量

- [ ] 所有经历使用 STAR 法则
- [ ] 至少 50% 的描述有量化数据
- [ ] 人性化检查全部 PASS
- [ ] 无编造的经历/数据

### Stage 4 — ATS 优化检查

```
ATS 检查：
[PASS] 版块标题标准（教育背景、实习经历、项目经历、技能、竞赛荣誉）
[PASS] JD 关键词自然出现在内容中（匹配: X, Y, Z）
[PASS] 联系信息格式规范
[PASS] 无关键词堆砌
[PASS] 日期格式统一
[PASS] 无装饰性符号干扰解析
```

### GATE 4 — 最终验证

- [ ] 所有 ATS 检查通过
- [ ] 简历内容完整（无占位符）
- [ ] 版块顺序符合策略
- [ ] 用户已确认信息摘要

### Stage 5 — 输出保存

生成两种格式：

**Markdown 格式**（人类可读）→ `简历_{用户名}.md`

**JSON 格式**（结构化数据，供 Module 5 匹配使用）→ `简历_{用户名}.json`

```json
{
  "name": "姓名",
  "education": {
    "school": "学校",
    "major": "专业",
    "degree": "本科",
    "graduation": "2027.06",
    "tags": ["211", "双一流"]
  },
  "skills": {
    "languages": ["C/C++", "Java"],
    "tools": ["MySQL", "Burp Suite"],
    "other": ["渗透测试"]
  },
  "internships": [],
  "projects": [
    {
      "name": "项目名",
      "tech": ["Java", "Spring Boot"],
      "role": "全栈开发",
      "highlights": ["独立开发用户模块，支持 5000 并发"]
    }
  ],
  "awards": [
    {"name": "赛事名", "level": "华南赛区二等奖", "year": "2026"}
  ],
  "target_role": "安全工程师",
  "ats_keywords": ["渗透测试", "C/C++", "漏洞挖掘"]
}
```

### Stage 6 — 优化建议

生成简历后，主动给出 2-3 条针对性建议：

1. **经历充实方向** — 哪些描述可以进一步量化
2. **关键词补充** — JD 中出现但简历中缺失的关键词
3. **版块优化** — 是否需要增加/删减版块

## Rules

### 必须遵守
- ✅ 一次只问一个问题
- ✅ 用户确认信息摘要后再生成简历
- ✅ 用数据量化成果（"提升 30%" > "显著提升"）
- ✅ 保留用户原始表述，不擅自改写
- ✅ 对学校/学历如实标注，不做价值评判
- ✅ 输出 Markdown + JSON 双格式

### 禁止事项
- ❌ 编造用户的实习/项目经历
- ❌ 对学校做"好/差"的评价
- ❌ 一次性列出所有问题
- ❌ 跳过确认环节直接生成
- ❌ 使用"负责""参与""协助"等弱动词
- ❌ 使用"精通""扎实""丰富"等空洞形容词

## Common Rationalizations

| 借口 | 现实 | 应对 |
|------|------|------|
| "用户信息太少，生成不了" | 信息少也可以生成基础版 | 生成后标注"建议补充"项 |
| "没有 JD，跳过分析" | 没有 JD 也要按行业标准优化 | 使用通用校招关键词 |
| "量化数据不好编" | 没有精确数据用范围/规模 | 用"管理 10+ 数据表""服务测试团队"等 |

## Verification

- [ ] 简历包含所有必要模块
- [ ] 所有经历基于用户真实信息
- [ ] 用户已确认信息摘要
- [ ] Markdown + JSON 文件已保存
- [ ] 路径已告知用户
- [ ] 优化建议已给出
