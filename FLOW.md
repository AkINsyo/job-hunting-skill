> 定义 5 个模块之间的依赖关系、调用顺序和数据流转。

## 数据依赖关系

| 模块 | 输入 | 输出 | 依赖 |
|------|------|------|------|
| Module 1 简历生成 | 用户口述 / 简历文件 | `简历_{用户名}.md` | 无 |
| Module 2 校招查询 | 公司名列表 | `校招网站汇总.md` | 无 |
| Module 3 岗位拉取 | 公司名 + 校招URL | `jobs/{公司}_岗位.json` + `.md` | Module 2（可选） |
| Module 4 简历匹配 | 简历 + 岗位JD | `匹配结果_{日期}.md` | Module 1 + 3 |
| Module 5 差距分析 | 简历 + 岗位JD | `提升计划_{日期}.md` | Module 1 + 3 |

## 三种调用模式

### 模式 A：独立调用

用户只触发单个模块，模块之间无依赖。

```
用户：帮我写简历          → 仅执行 Module 1
用户：找小米的校招网站     → 仅执行 Module 2
用户：看看字节在招什么     → 仅执行 Module 3
```

### 模式 B：链式调用

按依赖关系依次执行。

```
用户：帮我匹配岗位
  → 检查 → 无简历 → 先执行 Module 1
  → 检查 → 无JD   → 先执行 Module 2 → Module 3
  → 执行 Module 4
  → 询问是否继续 Module 5
```

### 模式 C：完整流程

用户要求从头开始，依次执行全部模块。

```
用户：完整流程 / 帮我从头开始

  Step 1: Module 1 — 采集信息，生成简历
       ↓ 用户确认简历
  Step 2: Module 2 — 查找目标公司校招网站
       ↓ 用户确认公司列表
  Step 3: Module 3 — 拉取岗位 JD
       ↓ 保存到 jobs/ 目录
  Step 4: Module 4 — 匹配分析
       ↓ 输出匹配结果
  Step 5: Module 5 — 差距分析（询问用户是否需要）
       ↓ 输出提升计划
  完成：告知用户所有产出文件位置
```

## 触发词 → 模块映射

| 用户意图（触发词） | 目标模块 | 说明 |
|-------------------|---------|------|
| 帮我写简历 / 生成简历 / 我没有简历 | Module 1 | 从零生成或解析已有简历 |
| 找校招网站 / 校招官网 / XX校招在哪 | Module 2 | 查找校招入口 |
| 看看在招什么 / 拉取岗位 / 查职位 | Module 3 | 拉取 JD 到本地 |
| 帮我匹配 / 哪些职位适合我 / 推荐岗位 | Module 4 | 需要 Module 1 + 3 的产出 |
| 分析不足 / 提升建议 / 我还需要学什么 | Module 5 | 需要 Module 1 + 3 的产出 |
| 完整流程 / 从头开始 / 一条龙 | 1→2→3→4→5 | 全流程依次执行 |

## 模块内调用流程

### Module 1 简历生成

```
开始
  ↓
Q1: 基本信息（学校/专业/学历/毕业时间）
  ↓ 等用户回复
Q2: 技术栈（编程语言/框架/其他）
  ↓ 等用户回复
Q3: 实习经历（公司/岗位/时长/产出）
  ↓ 等用户回复
Q4: 项目经历（项目名/技术/角色/亮点）
  ↓ 等用户回复
Q5: 其他亮点（竞赛/论文/开源，可跳过）
  ↓
复述摘要 → 用户确认
  ↓
生成 Markdown 简历
  ↓
给出 2-3 条优化建议
  ↓
保存为 简历_{用户名}.md
  ↓
输出文件路径 → 询问是否继续
```

### Module 2 校招页面查询

```
开始
  ↓
解析输入：提取公司名列表
  ↓
查询已知域名库 → 命中？
  ├── 是 → 直接使用，跳到验证
  └── 否 ↓
搜索校招页面（web_search，最多3家并行）
  ↓
提取官方域名（排除第三方）
  ↓
curl 验证可访问性
  ↓
输出汇总表格
  ↓
保存为 校招网站汇总.md
```

### Module 3 岗位 JD 拉取

```
开始
  ↓
获取输入：公司名 + 岗位方向（可选）+ 城市（可选）
  ↓
获取校招URL → Module 2 已执行？
  ├── 是 → 使用已有 URL
  └── 否 → 调用 Module 2 逻辑获取
  ↓
搜索岗位（web_search，3轮不同关键词）
  ↓
对有价值的链接 web_fetch 抓取 JD
  ↓
提取结构化信息（company/position/skills/...）
  ↓
保存为 jobs/{公司}_岗位.json + .md
  ↓
输出汇总 → 告知文件位置
```

### Module 4 简历匹配

```
开始
  ↓
前置检查：简历存在？JD存在？
  ├── 缺简历 → 提示先执行 Module 1
  ├── 缺JD   → 提示先执行 Module 3
  └── 都有   ↓
加载简历数据 + 所有 JD 数据
  ↓
逐岗位匹配：
  学历（20%）+ 技能（40%）+ 经验（25%）+ 加分项（15%）
  ↓
按匹配度降序排列
  ↓
输出总表 + Top 3 详细分析 + 总结
  ↓
保存为 匹配结果_{日期}.md
  ↓
询问：是否需要差距分析？→ Module 5
```

### Module 5 差距分析

```
开始
  ↓
前置检查：简历存在？JD存在？
  ↓
聚合行业高频要求（统计JD中各要求出现频率）
  ↓
与简历对比 → 识别差距
  ↓
差距分级：🔴硬性门槛 / 🟡技能缺口 / 🟢加分项
  ↓
生成提升计划：
  P0 紧急（1-2周）→ P1 重要（2-4周）→ P2 锦上添花
  ↓
简历优化建议（突出/弱化/措辞修改）
  ↓
投递策略（优先级 + 节奏）
  ↓
保存为 提升计划_{日期}.md
```

## 文件输出清单

执行完整流程后，项目目录下会生成：

```
项目目录/
├── 简历_{用户名}.md              ← Module 1
├── 校招网站汇总.md               ← Module 2
├── jobs/
│   ├── {公司名}_岗位.json        ← Module 3（结构化）
│   ├── {公司名}_岗位.md          ← Module 3（可读版）
│   └── ...
├── 匹配结果_{日期}.md            ← Module 4
└── 提升计划_{日期}.md            ← Module 5
```

## 中途修改偏好

任何时候用户可以修改目标：

| 用户说 | 处理方式 |
|--------|---------|
| "换个方向" | 更新岗位方向，重新执行 Module 3 + 4 |
| "城市改一下" | 更新城市，重新执行 Module 3 + 4 |
| "加一家公司" | 增量执行 Module 2 + 3，更新 Module 4 |
| "重新生成简历" | 重新执行 Module 1，然后 Module 4 + 5 |

**原则**：只重新执行受影响的模块，不全部重跑。
