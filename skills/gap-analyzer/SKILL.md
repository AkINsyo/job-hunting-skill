# 差距分析 & 提升建议

## Overview

从已拉取的多个同方向岗位 JD 中聚合高频要求，与用户简历对比识别差距，按紧迫性和可提升性分级，生成个性化提升计划和简历优化建议。

## When to Use

### 适用场景
- ✅ Module 4 匹配完成后，用户想进一步了解不足
- ✅ 用户想知道"我还需要补什么"
- ✅ 用户想获得学习路径建议

### 不适用场景
- ❌ 用户没有简历 → 提示先执行 Module 1
- ❌ 没有岗位 JD 数据 → 提示先执行 Module 3
- ❌ 用户明确表示不需要提升建议 → 跳过此模块

## Core Process

### Phase 1: 前置检查

同 Module 4，确认简历数据和岗位 JD 数据均存在。

### Phase 2: 行业高频要求聚合

从已拉取的多个同方向岗位 JD 中，统计每项要求的出现频率：

```
出现频率 > 60% → 核心必备项
出现频率 30-60% → 常见要求
出现频率 < 30% → 加分项/小众要求
```

**输出表格**：

```markdown
## 📊 {岗位方向} — 行业高频要求

| 要求项 | 出现频率 | 你的掌握情况 | 状态 |
|--------|---------|-------------|------|
| Java/Go | 90% | Java 熟练 | 🟢 |
| MySQL | 85% | 有经验 | 🟢 |
| Redis | 75% | 无经验 | 🔴 |
| 分布式系统 | 65% | 无经验 | 🔴 |
| Docker/K8s | 45% | 了解 | 🟡 |
```

### Phase 3: 差距分级

将所有差距分为三级：

**🔴 硬性门槛（短期难以改变）**
- 学历不达标、专业不对口、毕业时间不匹配
- 建议：如实告知，放宽筛选条件或找不卡学历的公司

**🟡 技能缺口（可通过学习弥补）**
- 缺某项语言/框架/工具
- 每项给出：预计学习时间 + 推荐资源（免费优先）+ 练手项目

**🟢 加分项（锦上添花）**
- 竞赛、开源、博客等
- 给出短期可操作建议

### Phase 4: 生成提升计划

按优先级分阶段：

```markdown
## 🚀 你的提升计划

### P0 — 紧急（1-2 周内）
| 项目 | 行动 | 预计耗时 | 资源 |
|------|------|---------|------|
| Redis | 学习基础 + 做缓存项目 | 1 周 | Redis 官方文档 |

### P1 — 重要（2-4 周）
| 项目 | 行动 | 预计耗时 | 资源 |
|------|------|---------|------|
| 分布式基础 | 学习 CAP、微服务 | 1 周 | MIT 6.824 |

### P2 — 锦上添花（持续）
| 项目 | 行动 | 预计耗时 | 资源 |
|------|------|---------|------|
| 技术博客 | 每周一篇 | 持续 | 掘金/思否 |
```

**资源推荐原则**：
- 只推荐免费公开资源
- 优先官方文档 > 系统教程 > 零散博客
- 每项至少推荐 1 个可立即开始的资源

### Phase 5: 简历优化建议

根据目标岗位关键词：

```markdown
## 📝 简历优化建议

### 应突出的内容
- ✅ Java 项目经验，补充量化数据

### 可弱化/删除的内容
- ❌ 与目标岗位无关的经历

### 措辞修改建议
1. 「负责后端开发」→ 「独立负责用户模块，日均处理 10 万次请求」
2. 「使用 MySQL」→ 「设计并优化 MySQL，查询耗时降低 60%」
```

### Phase 6: 投递策略

```markdown
## 📮 投递策略

### 优先级
1. 🟢 高匹配 → 立即投递
2. 🟡 中匹配 → 补充 1-2 周技能后投递
3. 🔴 低匹配 → 保底，降低预期

### 节奏
- 每天投 3-5 家，不要一天投完
- 高匹配优先，利用新鲜感准备面试
- 投递后记录状态
```

保存为 `提升计划_{日期}.md`。

## Rules

### 必须遵守
- ✅ 高频要求必须基于实际拉取的 JD 统计，不能凭空列举
- ✅ 每个差距项必须给出具体可执行的行动
- ✅ 推荐资源必须免费且可公开获取
- ✅ 提升计划按优先级分阶段

### 禁止事项
- ❌ 推荐付费课程/服务
- ❌ 对用户的学校/学历做价值评判
- ❌ 给出无法验证的承诺（"学完这个就能进大厂"）
- ❌ 一次性推荐太多资源导致用户无从下手

## Common Rationalizations

| 借口 | 现实 | 应对 |
|------|------|------|
| "用户差距太大，建议也没用" | 差距大更要拆分步骤 | 分阶段，从最容易的开始 |
| "推荐的资源差不多" | 差不多 ≠ 一样 | 每项推荐至少 1 个具体资源 |
| "投递策略太简单了" | 简单但明确比复杂但模糊好 | 给出可直接执行的优先级 |

## Red Flags

- 🚩 所有差距都是 🔴 硬性门槛：建议用户重新评估目标
- 🚩 用户方向与岗位方向完全不匹配：建议调整方向
- 🚩 岗位数据太少（<3 个）：统计意义不足，建议先拉取更多
