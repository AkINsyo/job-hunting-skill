# 简历匹配 & 职位推荐

## Overview

将用户简历的结构化数据与已拉取的岗位 JD 逐项比对，从学历、技能、经验、加分项四个维度加权打分，按匹配度排序输出推荐结果。

## When to Use

### 适用场景
- ✅ 用户已有简历（Module 1 产出）且已有岗位 JD（Module 3 产出）
- ✅ 用户想知道哪些岗位最适合自己
- ✅ 用户想了解简历与岗位的具体差距

### 不适用场景
- ❌ 用户没有简历 → 提示先执行 Module 1
- ❌ 没有岗位 JD 数据 → 提示先执行 Module 3
- ❌ 用户只想看岗位列表不想要匹配分析 → 直接读取 jobs/ 目录

## Core Process

### Phase 1: 前置检查

执行前必须确认：

1. **简历数据是否存在？**
   - 检查 `简历_*.md` 文件
   - 不存在 → 提示用户：「还没有简历数据，请先执行简历生成（Module 1），或者直接告诉我你的学校、专业、技能栈。」

2. **岗位 JD 是否存在？**
   - 检查 `jobs/*.json` 文件
   - 不存在 → 提示用户：「还没有岗位数据，请先执行岗位拉取（Module 3），告诉我你想查哪些公司。」

### Phase 2: 加载数据

1. 读取简历结构化数据（从 `简历_*.md` 提取）
2. 读取 `jobs/` 目录下所有 `.json` 文件
3. 合并所有岗位到统一列表

### Phase 3: 多维度匹配

对每个岗位，从以下维度打分：

| 维度 | 权重 | 评分规则 |
|------|------|---------|
| 学历匹配 | 20% | 满足/超出 = 满分，不满足 = 0 |
| 技能匹配 | 40% | 逐条匹配 JD skills vs 简历技能，计算覆盖率 |
| 经验匹配 | 25% | 实习/项目经验与 JD 描述的相似度 |
| 加分项 | 15% | 满足加分项的数量占比 |

**综合匹配度** = 各维度加权得分，百分比表示。

**匹配标记**：
- 🟢 **满足**：简历中明确具备
- 🟡 **部分满足**：有相关但不完全匹配（如 JD 要"精通"但简历写"了解"）
- 🔴 **不满足**：硬性不达标或明显缺失

### Phase 4: 排序与筛选

按匹配度降序排列。分为三档：
- 🟢 高匹配：> 75%
- 🟡 中匹配：50% - 75%
- 🔴 低匹配：< 50%

### Phase 5: 输出

**总表**：

```markdown
# 📊 岗位匹配结果

> 基于简历：{简历文件名}
> 匹配岗位数：{N} 个

| 排名 | 公司 | 岗位 | 城市 | 匹配度 | 关键优势 | 关键缺口 |
|------|------|------|------|--------|---------|---------|
| 1 | 小米 | 后端开发 | 北京 | 🟢 85% | Java熟练 | 缺微服务 |
| 2 | 字节 | Go开发 | 北京 | 🟡 70% | Go基础 | 缺分布式 |
```

**Top 3 详细分析**：

```markdown
## 🏆 Top 1：小米 — 后端开发（匹配度 85%）

### 匹配详情
| 要求项 | JD 要求 | 简历情况 | 匹配 |
|--------|---------|---------|------|
| 学历 | 本科及以上 | 本科 | 🟢 |
| 语言 | Java/Go | Java 熟练 | 🟢 |
| 数据库 | MySQL、Redis | MySQL 有经验 | 🟡 |
| 分布式 | 了解微服务 | 无经验 | 🔴 |

### 推荐理由
- 核心技术栈高度匹配
- 有相关实习经历
- 缺口可通过短期学习弥补
```

**总结**：

```markdown
## 📋 总结

- 🟢 高匹配（>75%）：{n1} 个
- 🟡 中匹配（50-75%）：{n2} 个
- 🔴 低匹配（<50%）：{n3} 个

### 推荐投递顺序
1. 公司 — 岗位（匹配度最高）
2. ...
```

保存为 `匹配结果_{日期}.md`。

## Rules

### 必须遵守
- ✅ 每个匹配项必须展示具体的「JD 要求 vs 简历情况」对比
- ✅ 匹配度计算必须有明确的维度和权重
- ✅ Top 3 必须展开详细分析
- ✅ 不要编造简历中没有的能力

### 禁止事项
- ❌ 对"能不能进"做预判性结论
- ❌ 只给匹配度数字不给具体分析
- ❌ 跳过前置检查直接匹配
- ❌ 建议用户付费购买课程

## Common Rationalizations

| 借口 | 现实 | 应对 |
|------|------|------|
| "简历信息不够，匹配不准" | 信息有限也要基于已有数据分析 | 标注置信度，建议补充信息 |
| "岗位太多，挑几个看看" | 挑选会引入偏见 | 全量匹配，让用户自己选 |
| "匹配度差不多，不用排了" | 用户需要明确的优先级 | 即使差距小也要排出顺序 |

## Red Flags

- 🚩 所有岗位匹配度都很低（<50%）：建议用户调整目标方向
- 🚩 简历数据与 JD 数据方向完全不匹配：告知用户并建议重新采集
- 🚩 只有 1-2 个岗位数据：建议先拉取更多岗位

## Examples

### 示例：匹配后端开发岗位

**输入：**
- 简历：中南大学本科，Java/MySQL/Redis，1段实习
- 岗位：3个后端开发 JD（小米/字节/美团）

**输出：**
```
排名1: 小米-后端开发 | 🟢 82% | Java熟练匹配 | 缺微服务经验
排名2: 美团-后端开发 | 🟡 68% | MySQL匹配 | 缺分布式经验
排名3: 字节-Go开发  | 🟡 55% | Go不匹配 | 需转语言
```

- [ ] Top 3 有详细展开分析
- [ ] 匹配结果文件已保存
- [ ] 投递优先级已给出
